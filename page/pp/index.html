プロトタイプ汚染を理解したかった

2022-05-05

プロトタイプ汚染とは？

ざっくりと簡単にプロトタイプ汚染を説明すると、JavaScript(以下js)のオブジェクトに存在する__proto__というプロパティをつたって 任意のプロパティを追加・上書きしてオブジェクトの振る舞いを変える攻撃手法のことです。
この手法を理解するためには、jsのプロトタイプチェーンに対する理解が必要なのでまずこちらから説明したいと思います。

プロトタイプとプロトタイプチェーン

以下の簡単なクラスなどが書かれているコードがあります。

class Fag{
    constructor(name){
        this.name = name
    }
}

class AnotherFag extends Fag{
    constructor(name, modelName){
        super(name)
        this.modelName = modelName
    }
}

let bazelard = new Fag('Bazelard')
let zelfikar = new AnotherFag('Zelfikar', 'bazelard')

このコードにはFagクラスとそれを継承したAnotherFagクラスとそれぞれをインスタス化したbazelardオブジェクトとzelfikarオブジェクトが存在します。
そしてこのbazelardとzelfikarオブジェクトには__proto__という謎のプロパティが既に設定されています。

__proto__とは

__proto__の正体は自身の継承元や生成元となるオブジェクトのprototypeへのゲッター・セッターです。

bazelard.__proto__ === Fag.prototype //true
zelfikar.__proto__ === AnotherFag.prototype //true

図解するとこんな感じです。(FagクラスとAnotherFagクラスの__proto__は省いています。あと分かり辛かったらすみません。)

プロトタイプチェーンとは

プロトタイプチェーンはオブジェクトのプロパティやメソッドを探すときに__proto__内を辿っていき__proto__がnullになるまで探索されることをいいます。

bazelard.hogeを探すときの挙動はこんな感じです

1、bazelard直下にあるhogeプロパティを探す。見つからない。
2、bazelard.__proto__にあるhogeプロパティを探す。見つからない。//この__proto__が参照する先はFag.prototypeです
3、bazelard.__proto__.__proto__にあるhogeプロパティを探す。見つからない。//この__proto__が参照する先はObject.prototypeです
4、bazelard.__proto__.__proto__.__proto__を探索しようとするが__proto__がnullを返すので探索が終了

プロトタイプ汚染

では本題のプロトタイプ汚染に入りたいと思います。また、コードを以下のように改変してます。

class Fag{
    constructor(name){
        this.name = name
    }
}

Fag.__proto__.__proto__.hoge = 'test'
 
let bazelard = new Fag('Bazelard')
let gourai = new Fag('Gourai)
プロトタイプ汚染とは

ではもしFag.__proto__.__proto__.hogeに適当な値をセットし(上記7行目)、bazelard.hogeあるいはgourai.hogeを探した場合はどうなるのでしょうか？
結果は値を返します。

bazelard.hoge //test
gourai.hoge //test

また、Fag.__proto__.proto__の参照先はObject.prototypeなのでセットした値はほぼ全てのオブジェクトにも影響を与えます。

window.hoge //test
document.hoge //test

このように__proto__をつたって任意のオブジェクトを汚染することをプロトタイプ汚染と呼びます。

プロトタイプ汚染の発生原因

プロトタイプ汚染の発生原因となるコードはざっくりとすると以下のようなオブジェクトのプロパティを2つ以上、任意に指定できる場合に発生します。

let a = process.argv[2] //__proto__を想定
let b = process.argv[3] //なんでもいいです
let c = process.argv[4] //なんでもいいです
let obj = {}
obj[a][b] = c

console.log(({})[b])
ガジェット

攻撃に利用されるオブジェクトのことを「ガジェット」と呼びます。主に未定義のプロパティが狙われます。

成功しても攻撃が刺さらない例

プロトタイプ汚染ができただけでは潜在的な脆弱性にしかならず、ガジェットを見つけだすことで攻撃に繋がる脆弱性でもあります。

//ガジェットが存在しない例
Object.prototype.hoge = "1" //汚染に成功した

let code = "1 + 1"
eval(code)

この例ではガジェットがそもそも存在しないので汚染しても特に何もできないです。

//プロパティが初期化されている例
Object.prototype.isAdmin = "1" //汚染に成功した

user = {
    name: "Ao Gennai",
    isAdmin: false
}

if (user.isAdmin){
    //何かの処理
}

この例ではObject.prototype.idAdminよりuser.isAdminのほうが探索順位が上位なので攻撃は刺さらないです。

プロトタイプ汚染の見つけ方

フロントエンド側

フロントエンド側は主にXSSの実行に利用されており、 見つけ方は至って簡単でURLのパラメータやハッシュに__proto__[cat]=nekoや__proto__.dog=inuなどを付与して アクセスした後にコンソール画面でcatやdogなどが定義されているかをチェックするだけです。
ここら辺の脆弱なライブラリなどは有志の方がまとめて下さっているのでこちら 1をみてみるのがいいと思います。
また、electronだとXSSでのプロトタイプ汚染からRCEに繋がる恐れがあるので注意したほうがいいと思います。

バックエンド側

バックエンドは最悪RCEに発展することがありとても危険なところですが、実装次第なところがありこれといった見つけ方はないと思います。 RCEに持ってこれた例はknqyf263さんがいい感じにまとめ 2て下さっているのでこちら 2をみていただいた方がいいと思います。
(投げやりでごめんなさい)

終わりに

この脆弱性で攻撃する際にはプロトタイプ汚染を行う箇所とガジェット(汚染を行う対象)の両方を見つける必要があり、他の脆弱性よりも見つける手間がかかりますが、 XSSからRCEなど様々な脆弱性につなげることが出来るという汎用性が高く危険な脆弱性です。
また、認知度やこの脆弱性を理解することの難しさの相まってか、多数のwebサイト・アプリケーションにこの脆弱性が存在しXSSやRCEができたという研究 3結果 4があります。
